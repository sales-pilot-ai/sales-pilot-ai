# Sales Pilot AI（GAS版）— プロジェクト概要

Google Apps Script上で動く、営業担当者がブラウザだけで使える社内向けWebアプリ。Node.js CLI版（リポジトリルートの`PROJECT.md`参照）の機能をベースに、Google Workspaceドメイン内の社員がログインするだけで使える形へ移行したもの。

---

## 設計思想

CLI版と同じく、以下の原則を引き継ぐ（詳細background: `../docs/adr/ADR-001-google-sheets-as-source-of-truth.md`）。

1. **Google Sheetsを正とする**。営業リスト・企業分析・テンプレートは全てSheetsに保存し、追加のデータベースは持たない
2. **秘密情報はコード・Sheetsに置かない**。APIキー・担当者連絡先等はすべて`PropertiesService`（スクリプトプロパティ）経由で管理し、`設定`画面（Sprint7⑥）からアプリ内で参照・変更できる
3. **Router / Service分離**。`Router.js`は画面から`google.script.run`で呼ばれる薄い公開関数のみを持ち、実処理は`Services/*.js`に委譲する
4. **ヘッダー駆動型Sheets連携**。CLI版と同様、シートの列は日本語ヘッダー文字列で特定し、列の並び替え・追加に強い設計とする

## アーキテクチャ

```
┌───────────────────────────────────────────────┐
│  ブラウザ（社員のPC・Googleアカウントでログイン済み）  │
│  Web App（HtmlService / SPA構成、Views/*.html）       │
└───────────────────┬───────────────────────────┘
                    │ google.script.run（非同期・JSON）
┌───────────────────▼───────────────────────────┐
│  Google Apps Script（V8ランタイム）                   │
│  Code.js  … doGetエントリポイント・画面(page)⇄HTML  │
│  Router.js … 画面から呼ばれる公開関数を集約           │
│  Services/ … 業務ロジック（検索/送信/集計/CRUD等）    │
│  Config/   … Sheets列定義・スクリプトプロパティ       │
└──┬────────────┬────────────┬────────────┬──────┘
   │            │            │            │
┌──▼──┐   ┌────▼─────┐  ┌───▼────┐  ┌────▼─────┐
│Sheets│   │  Gmail    │  │ Maps/   │  │Properties │
│App   │   │App        │  │Gemini   │  │Service    │
└──────┘   └───────────┘  └────────┘  └───────────┘
```

- **フロントエンド**: `HtmlService`によるSPA。画面遷移はクライアント側JS（`Views/Index.html`の`navigateToPage()`）で`window.open(execURL + '?page=xxx', '_top')`により行う（IFRAMEサンドボックス環境での相対リンク問題を回避するため）
- **バックエンド**: Apps Script（V8）。`Router.js`が画面からの唯一の窓口
- **データストア**: Google Sheets（`営業リスト`・`企業分析`・`営業メールテンプレート`・`返信テンプレート`の4タブ）のみ
- **認証**: Apps Script Web Appの標準機能（ドメイン内ログイン強制）
- **通知・ローディング**: 共通トースト（`Views/Index.html`の`showToast()`）・ボタンスピナー（`.button--loading`）・プログレスバー（`.progress-bar`）を各画面が共通利用する

詳細な画面一覧・API・データモデルは`SPEC.md`を参照。

## 既知の設計上の制約・注意点

`../docs/webapp-migration-design.md`に記載の制約は引き続き有効。特に以下は実装上も意識する必要がある。

- **Apps Script 1実行あたり最大6分**。大量送信・大量検索は将来的に分割継続実行が必要になる可能性がある（現状の運用規模では未対応）
- **同時編集**: Sheetsの行単位更新（ステータス変更等）は`LockService`で排他制御する。単純なスクリプトプロパティの参照・更新（設定画面）は排他制御不要（1呼び出しごとにアトミック）
- **送信失敗の非永続化**: `SEND_STATUS.FAILED`（送信失敗）はSheetsへ書き込む処理が現状存在しない（`createSendMailForCompany_`は失敗時にthrowするのみ）。そのためDashboard/レポートの「送信率」「送信成功率」は「送信可否○の対象企業のうち実際に送信できた割合」として定義しており、「送信試行に対する成功率」ではない点に注意（詳細は`SPEC.md`のKPI定義参照）
- **営業メール生成時のテンプレート選択は未実装**: テンプレート管理画面（Sprint7⑤）で複数テンプレートを管理できるが、実際の営業メール生成（`generateSalesDraft`）は常にデフォルトテンプレートを使用する。生成時に非デフォルトテンプレートを選ぶUIは将来の拡張候補

## 次期構想（Version2 / スコープ外）

以下は今回のSprint6〜7の開発スコープには含めず、実装しない。将来検討する場合は改めて設計から行う。

- Google Maps自動営業リスト取得（定期実行によるリード自動収集）
- AI営業優先順位付け（leadScoreの活用等）
- 自動フォローアップ（リマインド通知の自動送信）
- Slack通知連携
- LINE通知連携
- AI営業アシスタント（対話型の営業支援）
- 詳細レポート（コホート分析等）
- 成約分析（成約要因の分析）
- 営業メール生成時の複数テンプレート選択UI
- 送信失敗のSheets永続化・厳密な送信試行成功率の算出
