# Sales Pilot AI — Google Apps Script Web化 設計書

**ステータス**: Draft（設計レビュー中・実装未着手）
**日付**: 2026-07-07
**対象**: 現行 Node.js CLI 版 → Google Apps Script ベースの社内向けWebアプリへの移行

## 前提条件（再確認）

- 社員がブラウザだけで利用できること
- Googleアカウントでログインする
- Google Apps Script を中心に構築する
- データは Google Sheets で管理する
- メールは Gmail を利用する
- 企業検索は Google Maps API を利用する
- 追加のサーバー**ホスティング**費用を発生させない（Google Maps APIの利用量に応じた課金は既存の前提として発生し得る。⑮②参照）
- 社内利用（Google Workspace ドメイン内）を前提とする

## 移行前に必ず確認すべき制約（先出し）

設計の前提として、現行CLIの一部機能はApps Scriptの実行環境ではそのまま移植できません。先に共有します。

| 現行機能 | 制約 | 対応方針（案） |
|---|---|---|
| Webサイト解析（`crawler/website-analyzer.js`、Playwright使用） | Apps Scriptはヘッドレスブラウザ・任意プロセス実行不可 | Web化フェーズでは対象外とする。`UrlFetchApp`で取得できる範囲（HTML取得＋簡易パース）に縮退、または将来的に別サービス化を検討 |
| 1回の実行時間 | Apps Scriptは1実行あたり最大6分（Workspaceでも同じ） | 大量送信・大量検索は「時間主導トリガーでの分割継続実行」を前提に設計する（⑨で詳細） |
| Gmail送信件数 | Workspaceアカウントで1日あたり上限あり（無償版は少ない） | 送信画面に「本日の残り送信可能数」を表示し、上限超過を事前に防ぐ設計とする |
| 同時編集 | 複数人が同時に営業リストを操作する可能性 | `LockService` で排他制御を行う（⑧で詳細） |

以下、①〜⑨で設計をまとめます。

## ① システム全体構成

```
┌─────────────────────────────────────────────┐
│  ブラウザ（社員のPC・Googleアカウントでログイン済み）  │
│  Web App（HtmlService / SPA構成）                    │
└───────────────────┬───────────────────────────┘
                    │ google.script.run（非同期・JSON）
┌───────────────────▼───────────────────────────┐
│  Google Apps Script（V8ランタイム）                   │
│  ・doGet(e) … Web Appエントリポイント                  │
│  ・Router層 … 画面から呼ばれる関数を集約                │
│  ・Service層 … 業務ロジック（検索/送信/集計/etc.）       │
└──┬────────────┬────────────┬────────────┬──────┘
   │            │            │            │
┌──▼──┐   ┌────▼─────┐  ┌───▼────┐  ┌────▼─────┐
│Sheets│   │  Gmail    │  │ Maps   │  │Properties │
│App   │   │App/MailApp│  │(UrlFetch)│ │Service    │
└──────┘   └───────────┘  └────────┘  └───────────┘
   │            │              │
Google Sheets  Gmail      Google Maps
(営業リスト等)          Places API (New)
```

- **フロントエンド**: `HtmlService` によるSPA。画面遷移はクライアント側JSで切り替え、サーバー呼び出しは `google.script.run` に統一する。
- **バックエンド**: Apps Script（V8）。Router層とService層を分離し、CLI版の `src/cli/commands/*` → `src/sheets/*` / `src/gmail/*` / `src/providers/*` という現行の責務分割をそのまま引き継ぐ。
- **データストア**: Google Sheets のみ。追加DBを持たない（前提条件どおり）。
- **認証**: Apps Script Web Appの標準機能でGoogleログインを強制する（詳細は⑧）。
- **サーバーコスト**: Apps Scriptの実行・Sheets保存はいずれも無料枠内（Google Workspace契約内）で収まり、追加のホスティング費用は発生しない。

## ④ Google Sheets設計

1つのスプレッドシート内に、以下のタブ（シート）を持つ構成とする。現行CLI版のスキーマを基本的に継承し、Web化にあたり利用者・権限を管理する「権限」シートを新設する。

### シート一覧

| # | シート名 | 役割 | 主キー |
|---|---|---|---|
| 1 | 営業リスト | メインデータ（現行と同一） | 企業ID |
| 2 | 送信履歴 | 送信ログ（追記専用） | 送信日時 + Batch ID（複合） |
| 3 | 返信履歴 | 返信ログ（追記専用） | 検知日時 + 企業ID（複合） |
| 4 | テンプレート（新規） | メールテンプレート本体 | テンプレート名 |
| 5 | 設定（新規） | 秘密情報以外の運用設定 | キー |
| 6 | 権限（新規） | Webアプリの利用者・ロール管理 | メールアドレス |
| 7 | 営業ダッシュボード（任意で維持） | Sheets単体で見たい人向けの読み取り専用集計 | - |

### カラム一覧

**① 営業リスト**（現行と同一・19列）

`企業ID, 会社名, 業種, エリア, ホームページ, メールアドレス, お問い合わせフォーム, 電話番号, 住所, メモ, 送信日, 送信可否, 送信状況, 担当者名, 最終更新, 返信有無, 商談日, 成約, Place ID`

- 送信可否: `○` / `×` / （空）
- 送信状況: `未送信` / `送信済` / `送信失敗` / `返信あり` / `配信停止`
- 成約: `成約` / `失注` / （空）

**② 送信履歴**（現行と同一・追記専用・14列）

`送信日時, Batch ID, 企業ID, Place ID, 会社名, メールアドレス, 件名, Message ID, 送信結果, エラー内容, 送信者, Template Name, Scenario Name, Sales Pilot Version`

- 送信結果: `SUCCESS` / `FAILED` / `SKIPPED`

**③ 返信履歴**（現行と同一・追記専用・12列）

`検知日時, 企業ID, Place ID, 会社名, メールアドレス, 送信Message ID, 返信Message ID, 返信日時, 返信者, 件名, 内容概要, 返信要約`

**④ テンプレート**（新規）

`テンプレート名, 表示名, 件名, 本文（テキスト）, 本文（HTML）, 説明, デフォルトフラグ, 作成日, 更新日, 更新者`

- テンプレート名は `^[a-z][a-z0-9_]*$` 形式（現行バリデーションを継承）
- 本文中は `{{companyName}} {{contactName}} {{meetingUrl}} {{introText}}` をプレースホルダーとして展開（現行と同一）

**⑤ 設定**（新規）

`キー, 値, 説明`

- 例: `senderName`（送信元表示名）, `meetingUrl`（面談用URL）, `followUpWarningDays`（既定3）, `followUpUrgentDays`（既定7）, `defaultTemplate`（既定`initial_contact`）, `dailySendLimit`（Gmail送信上限の目安値）

**⑥ 権限**（新規）

`メールアドレス, 表示名, ロール, チームID, 追加日, 追加者, 有効フラグ`

- ロール: `管理者` / `一般社員` / `営業責任者`
- チームIDは現時点では未使用（空欄可）だが、将来「営業責任者が自分のチームのみ閲覧できる」機能（⑭参照）をスキーマ変更なしで追加できるよう先行して列を用意しておく

**⑦ 営業ダッシュボード**（任意で維持）

`会社名, メールアドレス, 電話番号, 送信状況, 送信日, 返信有無, 商談日, メモ`（現行と同一の3セクション構成）

**テンプレートをファイル管理からSheets管理に変更する理由**: Apps Scriptには永続的なローカルファイルシステムがなく、「データはGoogle Sheetsで管理する」という前提条件にも合致するため、`templates/emails/*.txt` + `templates.json` の構成をテンプレートシートの行データに置き換える。

**APIキーなどの秘密情報はSheetsに置かない**（⑧で詳細、`PropertiesService`を使用）。

## ⑤ Apps Script構成

Router層とService層を分離し、現行の `src/` の責務分割を踏襲する。

```
Code.js                  … doGet(e) エントリポイント、画面(HTML)の出し分け
Router.js                … 各画面のHTMLから google.script.run で呼ばれる公開関数を集約
Services/
  SheetsService.js       … 営業リストのCRUD（現行 src/sheets/service.js 相当）
  MapperService.js       … ヘッダー⇄フィールドの変換（現行 src/sheets/mapper.js 相当）
  MapsService.js         … Places API 呼び出し（UrlFetchApp、現行 providers/google-maps.js 相当）
  MailService.js         … GmailApp送信・返信検知（現行 gmail/mailer.js, reader.js 相当）
  TemplateService.js     … テンプレートCRUD（現行 templates/manager.js 相当）
  FollowUpService.js     … フォローアップ優先順位ロジック（現行 sheets/follow-up.js 相当）
  ReportService.js       … 集計処理（現行 report コマンド相当）
  HistoryService.js      … 送信履歴/返信履歴の追記（現行 sheets/history-service.js 相当）
Models/
  Company.js             … 企業データのバリデーション（現行 models/company.js 相当）
Triggers/
  CheckRepliesTrigger.js … 時間主導トリガーから呼ばれる返信確認バッチ
  SendContinuation.js    … 大量送信時の分割継続実行（⑨のリスク対応）
  BackupTrigger.js       … スプレッドシート全体を日次でDriveにコピーするバックアップ処理（⑲で詳細）
Config/
  Constants.js           … ステータス定数等（現行 constants/index.js 相当）
  Settings.js            … PropertiesServiceラッパー
Views/（HTMLファイル）
  Index.html             … SPAシェル（共通ヘッダー・ナビゲーション）
  Dashboard.html / Search.html / CompanyList.html / CompanyDetail.html /
  Send.html / Templates.html / Replies.html / FollowUp.html / Report.html / Settings.html
```

現行のCLIコマンド1つ＝Service層の1メソッド、という対応を保つことで、移行時のロジック突き合わせを容易にする。

## ⑦ 将来的な機能追加を考慮した拡張性

- **Service層を薄いRouterから分離**しているため、新機能（例：AIによる文面自動生成、Slack通知）は対応するServiceを1つ追加するだけで既存機能に影響を与えない。
- **テンプレートをSheets管理にした**ことで、将来AIが生成した文面もテンプレートシートの1行として自然に追加できる（ファイルシステムの制約を受けない）。
- **画面をSPA構成にした**ことで、新しい画面（例：Slack通知設定、権限管理）の追加が既存画面に影響を与えずに行える。
- **段階的な公開範囲の拡大**：Web AppのアクセスをGoogle Workspaceのグループ単位で制御できるため、まず営業チームのみに公開→軌道に乗ってから他部署へ展開、という段階展開が容易。
- **API化への拡張**：`doPost(e)` を追加すれば外部Webhook（Slack等からの起動）にも対応可能。将来「ロードマップ」で挙がっていたSlack通知機能はこの拡張点で実現できる。
- **AppSheet等への拡張**：将来的にモバイル対応が必要になった場合、同じSheetsをデータソースとしてAppSheetを併用することも可能（データ設計をSheets中心にしていることの利点）。

## ⑧ セキュリティ設計

- **ログイン強制**: Web Appのデプロイ設定を `Execute as: User accessing the web app` とし、アクセス範囲を `組織内のユーザーのみ`（Google Workspaceドメイン制限）に設定する。これにより社外アカウントでのアクセスを技術的に遮断する。
- **APIキーの管理**: Google Maps APIキーは `PropertiesService`（スクリプトプロパティ）に保存し、コード・Sheets上には平文で残さない。
- **OAuthスコープの最小化**: `appsscript.json` で要求するスコープを `spreadsheets`・`gmail.send`・`gmail.readonly` 相当に限定する（現行CLIのOAuthスコープと同等）。
- **データアクセス制御**: スプレッドシート自体の共有設定を「組織内のみ編集可」とする。将来的に画面側の操作権限を分けたい場合は、`Session.getActiveUser().getEmail()` で利用者を判定し、許可リスト（設定シート）に基づき機能を制限する拡張も可能。
- **排他制御**: 複数人が同時に営業リストを更新する可能性があるため、書き込み処理は `LockService.getScriptLock()` で排他制御し、更新の競合・データ破損を防ぐ。
- **破壊的操作の確認**: 削除・一括送信などの取り消せない操作は、画面側で確認ダイアログを必須化する（現行の「運用ルール」で人的ルールとしていた部分を、UIレベルの制約に格上げする）。
- **監査ログ**: 送信履歴・返信履歴はそのまま監査ログとして機能させる。将来的に「誰がいつ何を変更したか」の操作ログシートを追加することも検討可能。

## ⑨ 開発ロードマップ（MVPファースト）

「最初に最低限使える状態（MVP）を作り、その後に機能を積み上げる」順序とする。各PhaseはPhase 3終了時点で「検索→保存→送信」という営業の一連の流れがWeb単体で完結する状態（実用最小限）を目指す。詳細な画面・API対応は⑩⑬を参照。

| Phase | 内容 | 対象画面 | 主なAPI（⑬参照） | ゴール |
|---|---|---|---|---|
| Phase 0（現在） | 設計確定 | - | - | 本設計書のレビュー・承認 |
| **Phase 1** | ログイン | ①ホーム（枠のみ） | `getCurrentUser` | Web Appをデプロイし、Googleアカウントでログインしてダッシュボードの枠が表示される。権限シート未登録者には「権限が設定されていません」画面を表示するガードもここで実装する（⑪参照） |
| **Phase 2** | 営業リスト | ②企業検索、③営業リスト一覧、④企業詳細 | `searchCompanies` `saveCompanyCandidate` `listCompanies` `getCompanyDetail` `updateCompany` `reassignCompanyOwner` | 企業を検索して営業リストへ保存し、一覧・詳細で確認・編集できる。担当変更（管理者限定）もここで実装する |
| **Phase 3** | メール送信（MVP完成） | ⑤メール送信（3ステップ簡略版）、⑥テンプレートは参照のみ（CRUD画面なし） | `getSendableCompanies` `previewEmail` `sendEmails` `listTemplates` `getTemplate` | 検索→保存→送信までがWebで完結する（社内配布可能な実用最小限=MVP。⑦返信確認画面・⑨レポート・⑩権限管理UI・テンプレートCRUD画面・`getSendJobStatus`はMVPスコープ外とし後続Phaseへ先送り。⑮⑤参照） |
| Phase 4 | 運用支援機能・テンプレート管理UI | ⑦返信確認、⑧フォローアップ、⑨レポート、⑥テンプレート管理（CRUD）、①ダッシュボード充実 | `listReplies` `runCheckRepliesNow` `getFollowUpList` `getReportStats` `getDashboardSummary` `createTemplate` `updateTemplate` `duplicateTemplate` `deleteTemplate` `setDefaultTemplate` | 日々の運用（フォロー漏れ防止・成果の可視化）をWebで代替でき、テンプレートを画面から管理できる |
| Phase 5 | 権限・設定の充実 | ⑩設定（権限管理含む） | `listPermissions` `upsertPermission` `removePermission` `getSettings` `updateSettings` `getDiagnostics` | 複数人・複数ロールでの運用に対応し、管理者が設定変更・利用者管理を行える |
| Phase 6 | 社内試験運用・本格移行 | 全画面 | - | CLI版と並行運用しデータ整合性を確認、問題なければCLI版の運用終了を判断し全営業担当者へ展開 |

Phase 3では、大量送信時の分割継続実行（1実行6分制限への対応）を優先的に検証する。Phase 6では、CLI版とWeb版で同じ営業リストを参照しながら並行稼働させ、集計結果の一致を確認したうえで本格移行に進む。

## ⑪ 権限設計

現行CLI版は単一ユーザー運用のため権限区分がなかったが、Web化により複数人が同時アクセスするため新設する。ロールは「権限」シート（④参照）で管理し、`Session.getActiveUser().getEmail()` で判定する。

### ロール定義

| ロール | 想定ユーザー | 概要 |
|---|---|---|
| 管理者 | システム管理・営業企画担当 | 全データ・全設定・利用者権限を管理できる |
| 一般社員 | 営業担当者 | 自分が担当する企業（`担当者名`列で判定）の営業活動を行える |
| 営業責任者（将来） | チームマネージャー | 自分のチーム（`チームID`列で判定）配下の全担当者データを閲覧・管理できる |

### 機能別権限マトリクス

| 機能 | 管理者 | 一般社員 | 営業責任者（将来） |
|---|---|---|---|
| ログイン・ダッシュボード閲覧 | ○ | ○ | ○ |
| 企業検索・営業リストへの保存 | ○ | ○ | ○ |
| 営業リスト一覧の閲覧 | 全件 | 全件（閲覧のみ） | 全件 |
| 企業詳細の編集（商談日・成約・メモ） | 全件編集可 | 自分が担当者の企業のみ編集可 | チーム配下の企業を編集可 |
| メール送信 | 全件送信可 | 自分の担当企業のみ送信可 | チーム配下の企業を送信可 |
| テンプレートの作成・編集 | ○ | ○ | ○ |
| テンプレートの削除 | ○ | × | ○ |
| 返信確認（手動再実行） | ○ | ○ | ○ |
| フォローアップ閲覧 | 全件 | 自分の担当企業のみ | チーム配下のみ |
| レポート閲覧 | 全社集計 | 自分の担当分の集計のみ | チーム集計 |
| 設定変更（送信元・面談URL・閾値等） | ○ | × | × |
| 利用者・権限管理 | ○ | × | × |
| データ削除（企業・テンプレート等） | ○ | × | △（要検討） |

- 「自分が担当者の企業」は営業リストの`担当者名`列とログインユーザーの表示名を突き合わせて判定する。表記ゆれを避けるため、将来的には`担当者名`をメールアドレスベースの列に置き換えることも検討する（未確定事項に記載）。
- 権限チェックはRouter層の各公開関数の先頭で共通のガード関数（例: `PermissionService.assertCanX()`）を呼ぶ形で実装し、画面側のボタン表示・非表示だけに依存しない（クライアント側の非表示はUXのためであり、サーバー側でも必ず再チェックする）。
- 営業責任者ロールは「チームID」列を今のうちに権限シートへ用意しておくことで、実装時にスキーマ変更が発生しないようにする（⑭参照）。
- **権限シートに未登録のユーザーへの対応（Phase1のスコープに含める）**: Web Appの公開範囲はGoogle Workspaceドメイン制限のみのため、ドメイン内であれば権限シートに未登録でもログイン自体は技術的に可能。`getCurrentUser()`が権限シートに該当行なしと判定した場合は、通常画面を表示せず「権限が設定されていません。管理者に連絡してください」という専用画面のみを表示し、以降のAPI呼び出しをすべて拒否する。これによって「社員追加を忘れていた場合に空の一般社員扱いで操作されてしまう」事故を防ぐ（⑰運用ルール「社員追加」参照）。

## ⑬ デザインルール

社内システムであることを踏まえ、以下を基本方針とする。

| 項目 | 方針 |
|---|---|
| 全体の印象 | シンプル・Google Workspace（Gmail／スプレッドシート）に近い見た目 |
| 背景 | 白背景を基調とする（`#FFFFFF`） |
| アクセントカラー | 濃い青（例: Google Blue系 `#1A73E8`前後）をボタン・リンク・選択状態のみに使用し、多用しない |
| 余白 | セクション間・カード内は広めに確保し、詰め込まない |
| アイコン | 最小限に留める（ナビゲーション・操作ボタンなど意味が一目でわかる箇所のみ）。装飾目的のアイコンは使わない |
| フォント | OS標準のシステムフォント（`Roboto` ／ `Noto Sans JP`相当）を使用し、独自フォントは読み込まない |
| コンポーネント | テーブル・カード・ボタンなど最小限の部品を共通化し、画面ごとに見た目を変えない |
| フィードバック | 操作結果（保存完了・送信完了・エラー）はトースト通知またはインライン表示で簡潔に伝える |
| レスポンシブ | PC幅を基本としつつ、768px未満はモバイル向けレイアウト（⑩各画面のスマホ対応欄を参照）に切り替える |

これらはHTMLファイル（`Views/`）で共通のCSS（`Views/Style.html`のような共通インクルード）として一元管理し、画面ごとに独自スタイルを増やさない。

## ⑭ 将来的な拡張性の確認

⑦で挙げられた将来機能について、現在の設計のままで対応できるかを確認する。

| 将来機能 | 現設計での対応可否 | 理由・対応方法 |
|---|---|---|
| AIメール生成 | **対応可**（設計変更不要） | テンプレート＋変数展開の仕組みをそのまま利用できる。個別生成文言（現行CLIの`personalizer`相当）は`PersonalizerService.js`を1つ追加し、`previewEmail()`が返す本文にAI生成結果を差し込むだけでよい。生成結果をテンプレートシートの1行として保存すればファイルシステム制約も受けない |
| Slack通知 | **対応可**（設計変更不要） | `Code.js`に`doPost(e)`を追加すればSlackからのWebhook受信、`UrlFetchApp`によるSlackへの通知送信いずれも可能。通知先設定は「設定」シートにキーを追加するだけで足りる |
| LINE通知 | **対応可**（設計変更不要） | LINE Messaging APIも`UrlFetchApp`で呼び出せるため、Slack通知と同様に`NotificationService.js`を1つ追加するだけで対応できる |
| 案件管理 | **対応可**（軽微な拡張で対応） | 現行の「営業リスト」1シートに商談・案件の詳細を持たせると列が肥大化するため、新規シート「案件」（企業IDを外部キーとして持つ）を追加する想定。Sheetsベースの設計のため新シート追加のみでよく、既存シートの構造変更は不要 |
| 分析画面 | **対応可**（設計変更不要） | レポート機能と同じく既存3シート（営業リスト・送信履歴・返信履歴）の読み取り集計で実現できる。画面はSPA構成のため、既存画面に影響を与えず`Analytics.html`を追加するだけでよい |

### 拡張性を支えている設計判断

- **Service層の分離**（⑤）: 新機能は対応するServiceを1つ追加するだけで既存機能に影響を与えない
- **テンプレートのSheets管理**（④）: AI生成結果もテンプレートシートの1行として自然に追加できる
- **設定シート・権限シートの汎用キー・バリュー／列追加のしやすさ**（④⑥⑪）: 新しい設定値・チーム管理を後から列追加のみで対応できる
- **SPA構成**（①）: 新画面の追加が既存画面に影響を与えない
- **`doPost(e)`拡張点**（⑦）: 外部サービス（Slack／LINE）からのWebhook受信・送信の両方に対応できる

以上より、⑦で挙げられた将来機能はいずれも**現行設計のスキーマ・アーキテクチャを変更せずに追加できる**ことを確認した。

## ⑮ セルフレビュー（SaaS化・運用視点）

Phase1着手前に、「将来SaaS化する可能性」を見据えて①〜④、および「まず社員が毎日使えるものを最速で公開する」観点で⑤を、現行設計に対して自己レビューする。

### ①複数会社（マルチテナント）での利用可否

**結論: 現行設計はマルチテナント非対応。1社（1つのGoogle Workspaceドメイン）専用の設計になっている。**

| 観点 | 現状 | 問題点 |
|---|---|---|
| データの区分 | 全シート（営業リスト・送信履歴・返信履歴・テンプレート・設定・権限）に「どの会社のデータか」を示す列が存在しない | 会社A・会社B・会社Cが同じスプレッドシートを使うと、企業データ・送信履歴・テンプレートがすべて同じ行に混在し、区別できない |
| ログイン制御 | Web AppのアクセスをGoogle Workspaceドメイン単位で制限する設計（⑧） | 「1ドメイン＝1社」を前提にしており、そもそも複数の異なるドメインのユーザーが同時に使うことを想定していない |
| 権限シート | メールアドレスとロールのみで判定（チームIDはあるが営業チーム内の分類であり会社間の分離ではない） | 会社Aの管理者と会社Bの管理者が同じロール名を持てば、互いの全データにアクセスできてしまう構造 |
| デプロイ単位 | 1つのApps Scriptプロジェクト＝1つのスプレッドシートに固定 | 複数社に提供するには「会社ごとにコピーして個別デプロイ（テンプレート配布）」が現実的な現状の限界で、これは「SaaS」ではなく「配布型ツール」に近い |

→ 現行設計は「社内専用ツールとして正しい」設計であり、これは前提条件（社内利用・Workspaceドメイン内が前提）とも一致している。**今回はSaaS化を目的とした設計変更は行わず、将来の拡張ポイントとして認識しておく**、という整理が妥当（詳細は④）。

### ②Googleだけで運用できるか

| サービス | 現行設計での利用 | 費用 |
|---|---|---|
| Google Apps Script | 実行基盤 | 無料（Workspace契約に含まれる） |
| Google Sheets | データストア | 無料 |
| Gmail | 送信・受信 | 無料（Workspace契約に含まれる） |
| Google Maps API（Places API (New)） | 企業検索 | **有料**（無料クレジット枠を超えると課金） |
| Google Drive | 明示的な利用なし（スプレッドシート自体はDrive上に置かれるが機能としては未使用） | - |

**追加サーバー・有料サービスが必要になる箇所一覧:**

1. **Google Maps Platform（Places API）** — 前提条件に「企業検索はGoogle Maps APIを利用する」と明記されている時点で、実は既に有料サービスに依存している。無料枠（月$200クレジット相当）を超える検索ボリュームになれば課金が発生する。前提条件の「追加のサーバー費用を発生させない」という記述は、正確には「追加のサーバー**ホスティング**費用は発生させないが、Maps APIの利用量に応じた課金は既存の前提として発生し得る」と修正すべき
2. **（将来）AIメール生成** — "Googleだけで"を維持するならGemini API／Vertex AIの利用が必要になり、これも利用量課金の対象（⑦⑭で触れた将来機能だが、"Googleだけで"という制約と初めて衝突する機能なので明記しておく）
3. **（将来）Slack／LINE通知** — Google範囲外の外部サービス
4. **（将来・大量送信時）** Gmail送信件数がWorkspaceの上限（後述③）に達する規模になった場合、SendGrid等の外部メール配信サービスへの切替が必要になる可能性

→ **MVP（社内利用）の範囲では、Maps APIの課金を除けば、Apps Script／Sheets／Gmailのみで運用できる。** Driveは現状不要。前提条件の文言はMaps APIのコストに関する記述を補足すべき（後述「今のうちに直すべき点」）。

### ③Apps Scriptの制限（実際の運用で問題になる可能性がある値）

Google公式ドキュメント（[Quotas for Google Services](https://developers.google.com/apps-script/guides/services/quotas)）に基づく主要な制限値と、本システムでの影響:

| 項目 | 制限値 | 本システムでの影響 |
|---|---|---|
| 実行時間 | 1実行あたり最大6分（consumer／Workspace共通） | 大量送信（送信画面）・大量検索（企業検索）で6分を超えるリスクがある。すでに⑨で「分割継続実行」を対応方針としているが、Phase3での実測検証が必須 |
| トリガーの合計実行時間 | 1日あたり最大90分（個々の実行は6分以内） | 返信確認トリガーを高頻度（例: 5分おき）で走らせると、1日の合計実行時間の上限に到達しうる。トリガー頻度は「15〜30分おき」程度が現実的な下限になる |
| URL Fetch呼び出し数 | consumer 20,000回／日、Workspace 100,000回／日 | Maps API検索・Webサイト解析（`UrlFetchApp`）がここを消費する。Workspaceなら通常の営業活動では余裕があるが、大量検索を繰り返すと消費が早い |
| Gmail送信数 | consumerアカウント 500件／日、Workspaceアカウント 2,000件／日（新規契約は500件からスタートし、$100の累計課金＋60日経過後に2,000件へ昇格）。宛先数ベースでは1日3,000件（社外は2,000件まで） | **送信は担当者個人のGmailを介するため、上限は「担当者単位」**。1人の担当者が1日に数百社へ一斉送信するような使い方では上限に近づく。すでに設計で「本日の残り送信可能数」表示を予定しているが、新規契約直後は500件/日である点も踏まえた表示・警告ロジックが必要 |
| 同時アクセス・同時実行 | 公式に明確な数値は非公開（実務上、数十リクエスト程度が目安とされる） | `LockService.getScriptLock()`による排他制御は、営業リスト更新の直列化を意味する。社内の数名〜数十名規模では問題にならないが、ロック待ちがタイムアウト（既定30秒程度）した場合の再試行UI／エラーメッセージの用意が必要 |
| シートの速度 | 明確なクオータはないが、行数・列数が増えるほど`getRange`／`getDataRange`のレスポンスが線形に悪化する（実務上、数千行までは快適、1万行を超えると数秒〜十数秒の遅延が出やすい） | 現行`listCompanies(filter)`等は「全件取得してサーバー側でフィルタ」する設計を想定している。営業リストが数万行規模に成長した場合、一覧画面・レポート・フォローアップの応答が遅くなるリスクがある |
| ユーザー数 | 上記のクオータはすべて「スクリプトプロジェクト単位（＝全社員共通）」でカウントされる | 社員数が増えるほど、上記URL Fetch・トリガー実行時間のクオータをユーザー全体で分け合うことになる。Gmail送信数だけはユーザー単位で別枠なので、送信量が多いほどWorkspace案件では実は有利 |

→ **現状の想定規模（社内・数名〜数十名の営業担当者、1日あたり数十〜数百件の送信）であれば、いずれの制限も現実的な運用範囲内に収まる。** ただし「大量送信の分割継続実行」（⑨で既に触れている）と「シートの行数増加への対応」は、実測検証が必要な項目として残る。

### ④将来SaaS化する場合に作り直しが必要な箇所

| 領域 | 再利用できる範囲 | 作り直しが必要な範囲 |
|---|---|---|
| ビジネスロジック（Service層） | 検索・送信・集計・フォローアップ判定などのロジック自体は再利用可能 | データアクセス部分（`SheetsService.js`等）がSheets APIに直接依存しているため、Repositoryパターンで分離していない場合は書き直しが必要になる |
| 画面（View層） | HTML/CSS/JS、画面遷移、操作フローはほぼそのまま再利用可能 | テナント切り替えUI・契約プラン管理UIなど、SaaS特有の画面は新規追加 |
| Apps Script API一覧（⑫） | 関数名・パラメータ・戻り値の形（インターフェース）は資産としてそのまま持ち越せる | 内部実装がSheetsからDBに変わっても、Router層のシグネチャを維持すればフロントは変更不要にできる |
| データストア | スキーマ（カラム構成）は概念モデルとしてそのまま使える | Google SheetsをテナントごとのDBとして使う構成には限界があり、真のマルチテナントSaaSにするならFirestore／Cloud SQL等の本格的なDBへの移行が必要。移行にはService層のデータアクセス部分の全面書き直しが伴う |
| 認証 | ログイン後のロール判定ロジック（権限マトリクス、⑪）の考え方は再利用できる | 「Workspaceドメイン制限」から「テナントをまたいだ複数ドメイン・複数契約者を受け入れる認証」への作り直しが必要（サインアップ・招待・テナント紐付けの新規実装） |
| 実行基盤 | - | Apps Script自体が「1スクリプト＝1プロジェクト」という制約を持つため、1つのデプロイで複数会社を収容する構成にはそもそも向いていない。本格的なマルチテナントSaaSを目指す場合、Google Cloud Functions／Cloud Run＋Firestore等への基盤移行（＝Apps Scriptからの実質的な載せ替え）を検討する規模になる |
| 課金・プラン管理 | - | Stripe等の決済連携、プラン別機能制限、利用量計測は現行設計に一切存在せず、完全新規開発になる |

→ **「まず社内ツールとしてApps Scriptで作る」という今回の判断自体は妥当。** ただし、これは「将来そのままSaaS化できる設計」ではなく、「ビジネスロジック・UI・API定義は資産として持ち越せるが、データストア・認証・実行基盤は本格的な作り直しになる」という前提で意思決定していることを、関係者間で認識合わせしておくべき。

### ⑤MVPで削除・先送りすべき機能

「社員が毎日使えるものを最速で公開する」というMVPの目的に立つと、日々の営業活動の核（検索→保存→送信→フォロー）に直結しない画面・機能は後回しにできる。

| 画面／機能 | 削除・先送りの理由 | MVP期間中の代替運用 |
|---|---|---|
| ⑥テンプレート管理画面（作成・編集・複製・削除UI） | 開発コストが高い割に、初期はテンプレートが1〜2種類あれば営業活動は成立する | テンプレートシートを直接手編集する。送信画面が参照する`listTemplates`／`getTemplate`のみ実装し、CRUD系API（`createTemplate`等）はPhase4以降に先送り |
| ⑦返信確認画面（専用の一覧画面） | 返信検知そのもの（トリガーによる自動実行）は必要だが、専用一覧画面がなくても、企業詳細・ダッシュボードへの反映があれば運用は成立する | 返信確認バッチはトリガーで裏側で自動実行し、結果は企業詳細・フォローアップに反映するのみとする。`listReplies`はPhase4以降に先送り |
| ⑨レポート画面 | 集計値はあれば便利だが、なくても営業活動そのものは止まらない | Phase4以降に先送り。急ぎ数値が必要な場合は「営業ダッシュボード」シート（Sheets関数によるFILTER集計）で代替できる |
| ⑩設定画面のうち権限管理UI | 社内の少人数運用であれば、初期は全員同じ権限（一般社員相当）で開始しても実害は小さい | 権限シート自体は用意しておくが、管理者による権限変更はシートを直接編集する運用とし、`listPermissions`／`upsertPermission`／`removePermission`はPhase5以降に先送り |
| ⑤メール送信の「本文を確認する」詳細プレビューステップ | 4ステップウィザードのうち、対象確認とプレビューを1画面に統合しても運用上支障はない | ステップを「テンプレート選択→対象確認＋プレビュー→送信実行」の3ステップに簡略化する |
| `getSendJobStatus`（送信ジョブの進捗確認API） | MVP規模の送信件数（1回数十〜百件程度）であれば6分の実行時間内に収まる想定であり、進捗確認UIは不要 | 送信は同期的に完了を待つ設計とし、分割継続実行が実際に必要になった時点（Phase3の実測検証で判明した場合）でPhase4以降に追加する |
| `getDiagnostics`（接続診断） | 利用者（営業担当者）が使う機能ではなく開発者向けのツール | 画面には出さず、開発者がApps Scriptエディタから直接実行・ログ確認する運用でよい |

→ 上記を反映すると、**Phase1〜3のMVPは「①ホーム（簡易）／②企業検索／③営業リスト一覧／④企業詳細／⑤メール送信（3ステップ簡略版）／⑥テンプレート参照のみ／⑧フォローアップ」の7画面・最小限のAPIに絞り込める**。⑦返信確認画面・⑨レポート・⑩権限管理UIはPhase4〜5に後ろ倒しすることで、公開までの開発工数を圧縮できる。

### レビュー結果まとめ

**この設計で問題ない点**
- Service層とRouter層の分離、Sheetsベースのデータ設計、フェーズ分けの考え方は妥当で、社内ツールとして最速で公開する目的に合致している
- 権限シートに「チームID」列を先行して用意している判断（⑪）は、将来の小規模な拡張（営業責任者ロール）に対しては有効
- 想定される社内規模（数名〜数十名、1日数十〜数百件の送信）であれば、Apps Scriptの各種クオータ（③）は現実的な運用範囲内に収まる

**今のうちに直すべき点**
1. 前提条件の「追加のサーバー費用を発生させない」という記述を、「追加の**ホスティング**費用は発生させないが、Maps APIの利用量に応じた課金は既存の前提として発生し得る」に修正する（②）

**Phase1へ進んでよいか**

## ⑰ 運用ルール

| ルール | 内容 |
|---|---|
| **同じ企業への重複送信** | 送信対象は「送信可否＝○、送信状況＝未送信」の企業のみを自動抽出するため、同一テンプレートでの重複送信は通常発生しない。異なるテンプレート（例: 初回案内の後にフォロー文面）で同一企業へ複数回送信する場合は、担当者は送信前に必ず④企業詳細の送信履歴タブを確認する。検索時点の重複（同一Place IDの企業を再度検索・保存しようとするケース）は`saveCompanyCandidate`が既存レコードを更新する形で吸収するため、営業リストに同じ企業が2行できることはない |
| **担当変更** | 担当者名の変更は**管理者のみ**が④企業詳細の「担当者を変更」（`reassignCompanyOwner`、管理者限定）から行う。一般社員が自分で担当を付け替えることはできない。変更した際の自動通知（Slack等）は現行スコープにはないため、変更後は口頭・チャットで旧担当者・新担当者へ周知することを運用ルールとして必須にする |
| **社員追加** | 管理者が権限シートにメールアドレス・表示名・ロール（初期は「一般社員」を既定推奨）を追加する（MVP期間中は権限管理UIがないため⑩設定シートを直接編集、Phase5でUI化）。**追加を忘れて未登録のまま新入社員がログインした場合は、⑪で定義した「権限が設定されていません」画面が表示され、機能は一切使えない**（誤って無権限のまま操作されることはない） |
| **社員削除（退職・異動）** | 担当していた企業（担当者名列）が残ったままにならないよう、**必ず「担当変更」を先に完了してから、権限シートの有効フラグをOFFにする**（削除ではなく無効化を推奨。過去の送信履歴・返信履歴の「送信者」列はそのまま履歴として保持されるため削除しても矛盾は生じない） |
| **テンプレート更新** | MVP期間中はテンプレートシートを直接編集する（Phase4でCRUD画面化）。編集時は`更新日`と`更新者`（今回追加した列）を必ず入力する。**デフォルトテンプレートの変更は全社の送信内容に影響するため、変更前に営業チーム内で事前共有する**。既に送信済みのメールの本文は変わらない（送信履歴には件名のみ保存され本文のスナップショットは保存しない設計のため、送信後にテンプレートを変えても過去の送信内容の遡及確認はできない点を認識しておく） |
| **エラー時** | 担当者はまず1回だけ操作をやり直す。2回連続で同じエラーが出た場合は、エラーメッセージのスクリーンショットを管理者に共有し対応を依頼する（一次対応のルール化）。個別の対応は⑱障害時の対応を参照 |

## ⑱ 障害時の対応

利用者（営業担当者）がその場でとるべき行動を中心に整理する。

| 障害 | 症状 | 利用者の行動 |
|---|---|---|
| **Apps Script停止**（Google側障害・デプロイ不整合等） | Web Appにアクセスできない、白画面・タイムアウトになる | ①ブラウザを再読み込みする　②[Google Workspace Status Dashboard](https://www.google.com/appsstatus/dashboard/)でGoogle全体の障害情報を確認する　③他の社員にも同じ症状が出ているか確認する（自分だけの問題か全社的な問題かを見極める）　④全社障害の場合は復旧まで通常のGmail・電話での営業活動に切り替える（Webアプリに依存しない業務継続手段を運用として明文化しておく）　⑤復旧しない場合は管理者へ連絡する |
| **Maps APIエラー**（クォータ超過・APIキー無効など） | ②企業検索の実行時にエラーメッセージが出る、または結果が常に0件になる | ①表示されたエラーメッセージを確認する（「該当する企業が見つかりませんでした」と「検索サービスに一時的な問題が発生しています」を区別できるようにService層でメッセージを分ける、実装時の要件とする）　②クォータ超過が疑われる場合はその日の検索を控え、翌日以降に再試行する　③繰り返しエラーになる場合は管理者に連絡し、⑩設定の接続診断（`getDiagnostics`）でAPIキーの状態を確認してもらう |
| **Gmail送信失敗**（個別送信エラー・送信上限到達） | 送信完了後のサマリーに「失敗: N件」と表示される、または「本日の残り送信可能数」が0で送信ボタンが押せない | ①失敗した企業は④企業詳細の送信履歴タブでエラー内容を確認する（例: 無効なメールアドレス→メールアドレスを修正し送信可否を見直す）　②送信上限到達の場合は当日はそれ以上送信せず翌日に持ち越す（画面の残り送信可能数表示で事前に気づける設計になっている）　③繰り返し失敗する場合は管理者に共有する |
| **Sheets障害**（保存失敗・同時編集競合・応答なし） | 保存・送信操作が「失敗しました」というエラーになる、または画面の反応がなくなる | ①画面を再読み込みし、再度操作する（`LockService`のロック待ちタイムアウトが原因の場合、再試行で解消することが多い）　②同じ企業を他の社員が同時に操作していないか確認する　③繰り返し発生する場合はスプレッドシートを直接開き、意図しない変更がないか目視確認する　④Sheets自体は高可用性のGoogleサービスであり完全障害は稀なため、Google Workspace Status Dashboardも併せて確認する |

**共通の推奨事項**:
- 障害の連絡先（社内の管理者・システム担当）を①ホームまたは⑩設定に常設のヘルプリンクとして明記する（現行設計にはこの導線がないため、Phase1〜2のいずれかで追加する）
- Webアプリが使えない間の業務継続手段（Gmail・電話での直接対応）をあらかじめ社内に明文化し、障害時に営業活動そのものが完全停止しないようにする

## ⑲ バックアップ設計

| 対象 | バックアップ方法 | 保持期間・復元方法 |
|---|---|---|
| **スプレッドシート全体**（営業リスト・送信履歴・返信履歴・テンプレート・設定・権限のすべてのシートを含む） | ①Google Sheets標準の「版の履歴」（Driveの変更履歴）が全編集を自動記録する（追加コスト・追加実装なし）。②これに加えて、時間主導トリガー（`Triggers/BackupTrigger.js`）で**日次**にスプレッドシート全体を`DriveApp.makeCopy()`でバックアップ用Driveフォルダにコピーする。ファイル名は`営業リスト_backup_YYYYMMDD`とする | 直近30日分のコピーを保持し、それより古いものは同トリガーで自動削除する（Drive容量の圧迫を防ぐ）。復元は、直近の誤操作は「版の履歴」から即時ロールバック、より大きな問題（シート構造の破壊等）はバックアップコピーからの手動復元、の2段階で対応する |
| **テンプレート** | テンプレートシートは上記のスプレッドシート全体バックアップに含まれるため個別のバックアップ処理は不要 | デフォルトテンプレートの文言は`Config/DefaultTemplates.js`にもフォールバック値としてハードコードしておく（テンプレートシートが誤って全削除されても、初回案内文だけは失われない安全策・初期化用途を兼ねる） |
| **設定** | 設定シートも同様にスプレッドシート全体バックアップの対象。加えて、Web App公開設定・OAuthスコープ等の`appsscript.json`はGitリポジトリ（`gas/appsscript.json`）でバージョン管理されているため、コードのバックアップ（Git）でカバーされる | `PropertiesService`に保存するAPIキー等の秘密情報は、そもそもSheetsに置かない設計のため上記バックアップの対象外。値そのものの自動バックアップの仕組みはないため、「APIキーを発行した際の発行元・管理担当者の記録を社内のパスワード管理ツール等に別途保管する」運用ルールで対応する |

**運用ルールとしての補足**: バックアップが実際に復元可能であることを最低年1回（またはPhase6の試験運用開始前に1回）実際に検証する「復元テスト」を実施する。バックアップを取得しているだけで復元手順が機能しない、という事態を避けるための最低限の確認とする。

## ⑳ 最終セルフレビュー結果（運用視点）

Phase1着手前の最終ゲートとして、「社員が毎日使う運用」を前提に⑯〜⑲を検証した。

**この設計で問題ない点**
- ⑯の1日の利用シナリオは、⑮⑤で絞り込んだMVP画面構成（返信確認画面なし・テンプレートCRUD画面なし・権限管理UIなし）でも問題なく完結する。専用画面を削った機能は、既存画面（①ホーム・④企業詳細）への反映で代替できることを実際のシナリオで確認できた
- ⑰の運用ルールは、既存のデータ構造（送信可否・送信状況・有効フラグなど）だけで大部分をカバーできており、追加のシステム機能なしに運用でルール化できる
- ⑲のバックアップは、Google Sheets標準の「版の履歴」に加えて日次コピーを取る設計とすることで、追加コスト・追加サービスなしに実装できる

**今のうちに直した点（本レビューで反映済み）**
1. ④企業詳細に「担当者変更」項目がなかったため追加し、⑫に`reassignCompanyOwner`（管理者限定）を追加した
2. 権限シートに未登録のユーザーがログインした場合の挙動が未定義だったため、「権限が設定されていません」専用画面を表示する仕様を⑪に追加し、Phase1のスコープに組み込んだ
3. テンプレートシートに「誰が変更したか」を追跡できる`更新者`列を追加した
4. スプレッドシート・テンプレート・設定のバックアップ方法（⑲）を新設し、`Triggers/BackupTrigger.js`を⑤のフォルダ構成に追加した

**今後（Phase1着手をブロックしない範囲）で対応すべき点**
- 障害時の連絡先（管理者・システム担当）を明示するヘルプ導線が画面上にまだない。Phase1〜2のいずれかで①ホームまたは⑩設定に追加する
- テンプレート変更時の事前共有や、社員削除時の担当変更の順序は「システムではなく運用ルール」で担保している部分であり、システム的な強制力はない。将来的に（Phase4以降）ワークフロー的な確認ステップをシステム側に持たせることも検討余地として残す

**Phase1へ進んでよいか**
→ **問題ない。** 本レビューで見つかったギャップはいずれも軽微で、かつ①権限未登録ガードはPhase1のスコープに組み込み済み、②担当者変更はPhase2（企業詳細実装）で対応、③バックアップトリガーはPhase1のログイン基盤構築と同時に仕込んでおくのが望ましい（データが増える前にバックアップの仕組みを先に用意する）、という位置づけであり、いずれもPhase1着手のブロッカーにはならない。**Phase1（ログイン画面開発）へ進んで問題ない。**
