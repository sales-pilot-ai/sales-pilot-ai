# generateManualSpreadsheet実行時のsetSharingエラー修正

## 背景・依頼内容

Apps Scriptエディタから`generateManualSpreadsheet`を実行したところ、
`Exception: Invalid argument: permission.value`（`Scripts/GenerateManual.gs:525`、
共有設定の`setSharing`呼び出し部分）でスクリプトが停止したとの報告。原因調査と修正、
既に作成された中途半端なスプレッドシートの後始末処理の追加が依頼された。

## 原因

`DriveApp.File.setSharing(DriveApp.Access.DOMAIN_WITH_LINK, DriveApp.Permission.VIEW)`は、
API呼び出し自体の書き方としては公式ドキュメントどおり正しい。ただし`DOMAIN_WITH_LINK`
（同一ドメイン内でリンクを知っている人は閲覧可）は、ファイルの所有者がGoogle Workspaceの
ドメイン付きアカウントである場合にのみ有効な設定であり、個人のGoogleアカウント
（`@gmail.com`等）が実行者の場合はファイルに紐づく「ドメイン」自体が存在しないため、
Drive API側がこの権限を作成できずエラーになる。本プロジェクトのApps Scriptプロジェクトの
所有者アカウントはGmailの個人アカウントであるため、このケースに該当していたと考えられる
（Apps Script環境の外からは実行アカウントの種別を直接確認できないため、確実な断定はできないが、
発生した実際のエラー内容と一致する）。

## 実施したこと（gas/src/Scripts/GenerateManual.js）

1. **共有設定のフォールバック化**（`shareManualSpreadsheet_(ss)`を新設）
   - まず従来どおり`DriveApp.Access.DOMAIN_WITH_LINK`を試みる（Workspaceドメインアカウントで
     実行された場合は、依頼どおりドメイン限定共有になる）
   - 失敗した場合のみ`DriveApp.Access.ANYONE_WITH_LINK`（リンクを知っている人は誰でも閲覧可）
     にフォールバックする。フォールバックした場合、共有範囲がドメイン限定より広がる旨と
     元のエラー内容を`Logger.log`に明記する
   - フォールバックも失敗した場合は、スプレッドシート自体は既に完成しているためクラッシュさせず、
     「手動で共有設定を行ってください」というログを残して関数を正常終了させる
   - いずれのケースでも例外を再スローしないため、シート内容の生成が完了していれば
     スプレッドシート自体は必ず作成され、URLが返る
2. **前回実行分の同名ファイルの後始末**（`trashExistingManualSpreadsheets_()`を新設、
   関数冒頭で呼び出し）
   - `DriveApp.getFilesByName('Sales Pilot AI 取扱説明書')`で同名の既存ファイルを検索し、
     見つかった場合は`setTrashed(true)`でゴミ箱へ移動する（完全削除ではなく、Drive側から
     復元可能な状態にする）
   - 見つからない場合は何もしない
   - **注意**: 名前一致で検索するため、過去に成功したファイルであっても同名であれば
     再実行のたびにゴミ箱へ移動される（「前回実行分の重複ファイルが残らないように」という
     依頼の趣旨に沿った設計）。過去の生成結果を残しておきたい場合は、実行前にファイル名を
     変更しておく必要がある
3. スプレッドシートのタイトル文字列を`MANUAL_SPREADSHEET_TITLE`定数として一箇所にまとめ、
   検索・作成の両方で同じ値を使うようにした（タイポによる不一致を防ぐため）
4. タブ構成・データ内容・色分け・目次等、他の生成ロジックは一切変更していない

## 実施した検証

1. `npm run lint` → エラー0件
2. `npm test` → 40ファイル/806テスト 全てPASS
3. Node vmによるシミュレーション（実際のGenerateManual.jsをそのまま実行）
   - `DriveApp.Access.DOMAIN_WITH_LINK`が失敗する環境を再現し、`ANYONE_WITH_LINK`へ
     フォールバックして関数がクラッシュせずURLを返すこと、フォールバックの経緯が
     Loggerに残ることを確認
   - `DOMAIN_WITH_LINK`が成功する環境（Workspaceドメインアカウント想定）では、
     従来どおりフォールバックが発生しないことを確認（既存の正常系への影響がないことの確認）
   - `DOMAIN_WITH_LINK`・`ANYONE_WITH_LINK`の両方が失敗しても、関数がクラッシュせず
     完了し、手動設定を促すログが残ることを確認
   - 同名の既存ファイルが2件存在する状態から実行し、両方が`setTrashed(true)`で
     ゴミ箱へ移動され、件数がログに残ることを確認
   - 同名の既存ファイルが無い場合は何もしない（クラッシュもログの誤表示もない）ことを確認
   - 既存の全タブ構成・データ内容・色分け・目次のシミュレーション（前回作成分）を
     修正後のファイルに対して再実行し、他の生成ロジックに一切影響がないことを再確認

### 検証の限界（Apps Script環境でしか確認できない点）

- Node vmのモックは「`setSharing()`が特定の引数で例外を投げる」という**想定される挙動**を
  再現したものであり、実際のGoogle Drive API・Apps Script環境上で本当に
  `DOMAIN_WITH_LINK`が失敗し`ANYONE_WITH_LINK`が成功するかどうかは、実際に
  Apps Scriptエディタから実行して確認する必要がある（このセッション内ではOAuthトークンの
  期限切れにより`clasp run`等でのライブ実行確認ができない制約が継続している）
- 「実行アカウントが個人のGoogleアカウントであるためDOMAIN_WITH_LINKが使えない」という
  原因の特定も、エラーメッセージと既知のAPI仕様からの推測であり、Apps Script側のログ・
  実際の実行アカウントの種別を直接確認したものではない
