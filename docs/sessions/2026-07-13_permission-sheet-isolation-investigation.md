# タスクD: 権限シート秘匿化の調査・中断

## 背景・依頼内容

社内の他メンバーも営業データのスプレッドシートを共有利用する予定だが、「権限」シートだけは
オーナー以外に見せたくない。まずWebアプリの実行モードを確認し、`USER_DEPLOYING`（オーナーとして
実行）ならスプレッドシート分離を実施、`USER_ACCESSING`（アクセスユーザーとして実行）なら
認証が壊れるため実装せず中断して代替案を報告する、という依頼だった。

## 調査結果

`gas/src/appsscript.json`を確認したところ、実行モードは以下の通り：

```json
"webapp": {
  "executeAs": "USER_ACCESSING",
  "access": "ANYONE"
}
```

**`USER_ACCESSING`（アクセスユーザーとして実行）であることを確認した。**

## 判断: 分離を実装せず中断

依頼の条件分岐通り、`USER_ACCESSING`のためスプレッドシート分離は実装していない。

### なぜ分離すると認証が壊れるか

`USER_ACCESSING`モードでは、`SpreadsheetApp.openById()`等のすべての操作が
「今アクセスしているユーザー自身のGoogleアカウントの権限」で実行される
（開発者の権限では実行されない）。

`requireUser_()`/`requireAdmin_()`（ひいてはログイン判定そのもの）は、Admin・User
どちらのユーザーがアクセスした場合でも、**必ず「権限」シートを読み取ってロールを判定する**
処理を経由する。つまり、ログインしようとしている本人自身が、その「権限」シートを
読み取れる権限を持っていなければ、そもそもログイン判定自体が失敗する。

権限シートをオーナーのみアクセス可能な別スプレッドシートに分離すると、
オーナー以外の一般ユーザーが権限シートを開く権限を持たなくなるため、
一般ユーザーが自分自身のロールを確認しようとした瞬間に権限エラーが発生し、
**ログインそのものが全員（オーナー以外）失敗する。**

### 実行モードをUSER_DEPLOYINGに変える案について

実行モードを`USER_DEPLOYING`（常にオーナー権限で実行）に変更すれば、権限シートを
分離してもログイン判定自体は成立する。しかし、この方式にはSprint8で決定した
「Google Workspaceログインを活用し、実際の送信はログイン中のGoogleアカウントの
Gmailから行う」という設計と**根本的に矛盾する**。`USER_DEPLOYING`では、Gmail送信・
Gmail検索を含むすべての処理が常にオーナー（開発者）のGoogleアカウントで実行されるため、
営業担当者ごとに自分のGmailから送信する、という現在の中核設計が壊れる
（全員分のメールが開発者のGmailから送信されることになる）。

このため、実行モードの変更も現実的な解決策ではないと判断した。

## 代替案（優先順、それぞれのメリット・デメリット）

### 代替案1（推奨）: 権限シートを「保護」＋「非表示」にする（技術的な強制力は限定的）
- 方法: 「権限」シートを右クリック→シートを保護→「特定のユーザーのみ編集可」に設定し、
  さらにシートを非表示にする
- メリット: 実装コストがほぼゼロ、既存の認証の仕組みに一切手を加えない
- デメリット: これは「編集の制限」と「UIから見えにくくする」だけであり、閲覧権限を持つ
  ユーザー（スプレッドシート編集者）が「非表示のシートを表示」を選べば中身は見えてしまう。
  **真の意味でのアクセス制御ではなく、誤操作防止のためのものと理解した上で採用する必要がある**

### 代替案2: チームメンバーの共有権限を「閲覧者」または「コメント可」に制限する
- 方法: 営業データのスプレッドシート自体の共有設定を、担当者には「閲覧者」権限のみ付与し、
  企業データの追加・編集は既存のWebアプリの画面上の操作のみで完結させる
  （Sheetsを直接編集させない運用にする）
- メリット: 少なくとも「シートを直接編集して権限シートを書き換える」リスクは下げられる
- デメリット: 「閲覧者」でもシートの中身（権限シートを含む）は見えてしまうため、
  「見せたくない」という目的自体は達成できない。あくまで편集保護のための代替案

### 代替案3（大掛かり・今回は未実施）: ログイン方式・送信方式そのものを見直す
- 方法: Google Workspaceドメイン限定デプロイ＋`USER_DEPLOYING`実行モードへ切り替え、
  Gmail送信は各担当者個人のGmailではなく、共有の送信用アカウント（もしくは
  ドメイン全体委任を設定したサービスアカウント）経由に変更する
- メリット: 権限シートを含むすべての内部データを完全にオーナーのみのアクセスに分離できる
- デメリット: Sprint8で確定した「各担当者自身のGmailから送信する」という設計を覆す
  大規模な変更になる。Google Workspace管理者権限での追加設定も必要になる可能性が高い。
  今回のタスクの範囲を大きく超えるため、実施するかどうかは別途相談が必要

## 結果・現状

コードの変更は一切行っていない（appsscript.jsonの実行モード確認のみ）。

## 未解決の課題・次のアクション

- 上記代替案のうちどれを採用するか、ユーザーの判断を待つ
- 代替案3を検討する場合は、送信方式の変更が既存の「営業担当者ごとの送信履歴・
  返信確認」機能全体に影響するため、別タスクとして仕様を洗い出す必要がある

Skill化候補: なし（GASのexecuteAsモードとデータ分離の制約に関する調査であり、
本プロジェクト固有の意思決定記録として残す）
