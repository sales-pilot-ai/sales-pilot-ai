# Sprint3 設計書 — AI企業分析・営業メールDM下書き生成

**ステータス**: 確定（実装着手前）
**日付**: 2026-07-08
**前提**: `docs/webapp-migration-design.md`（Web化設計書）、Sprint1・Sprint2実装済み

## 背景・目的

Sprint2までで「企業検索 → 営業リスト保存 → 一覧表示」という営業リスト構築の基本フローが完成した。当初計画では次Sprintを「企業詳細・編集画面」としていたが、営業自動化への貢献度を優先し、**Google Gemini APIを用いた企業情報解析・営業メールDM下書き生成**を先行して実装する。企業詳細・編集画面は後続Sprintへ先送りする。

## 確定した方針（レビューでの決定事項）

| # | 項目 | 決定内容 |
|---|---|---|
| ① | Geminiモデル | `gemini-2.0-flash` を採用する |
| ② | 下書きの扱い | 「採用後の送信」までは対象外。生成された件名・本文を営業担当がコピー&ペーストして利用する運用とする。実際の送信機能（Sprint4予定）との接続は改めて設計する |
| ③ | データの持ち方 | 詳細な解析結果は新規シート「企業分析」に保存する。あわせて営業リストシートに要約列を1列追加し、一覧からでも概要を把握できるようにする |
| ④ | 下書きの保存方式 | 履歴は持たず、直近1件のみ上書き保存するシンプルな方式とする |
| ⑤ | 画面構成 | 独立した専用画面は作らず、営業リスト一覧の各行から展開できる**インラインアコーディオン**として実装する |

## ① 画面設計

### 営業リスト一覧画面（`CompanyList.html`）の変更

既存のテーブルに以下を追加する。

- 各行に「AI分析」列（トグルボタン、例: `▶ AI分析` / `▼ AI分析`）を追加
- 「AI要約」列を追加し、営業リストシートの新列をそのまま表示する（未解析の場合は「未解析」と表示）
- 「AI分析」トグルをクリックすると、その行の直下に詳細パネル（`<tr>`1行、`colspan`で全列幅を使用）をインライン展開する

### インラインパネルの内容（展開時）

```
┌─────────────────────────────────────────────────────┐
│ 事業内容: ...                                          │
│ 特徴: ...                                              │
│ ターゲット層: ...                                       │
│ 強み: ...                                              │
│ キーワード: ...                                         │
│ 解析日時: 2026-07-08 10:00 / モデル: gemini-2.0-flash    │
│                                    [ 再解析を実行する ]  │
│                                                       │
│ 営業メール・DM下書き                                     │
│ トーン: [初回アプローチ ▼]         [ たたき台を生成する ] │
│ 件名（下書き）: [__________________________]（コピー） │
│ 本文（下書き）:                                         │
│ [_________________________________________]（コピー） │
│ 生成日時: 2026-07-08 10:05                             │
└─────────────────────────────────────────────────────┘
```

- 未解析の企業の場合は「解析を実行する」ボタンのみ表示し、結果表示欄は空にする
- 解析中・生成中はボタンを無効化しローディング表示にする
- 解析・生成が失敗した場合は、失敗理由をパネル内にエラーメッセージとして表示する（設計書要件⑤の考え方を踏襲）

### 営業リスト一覧のフィルタへの影響

今回のSprintでは「AI要約」列によるフィルタ機能は追加しない（表示のみ）。将来的にキーワード検索の対象に含める拡張は可能。

## ② データ構造（Google Sheets）

### 新規シート「企業分析」

企業1件につき1行（企業IDが主キー、再解析・再生成時はUPDATE=upsert）。

| 列 | 内容 | 備考 |
|---|---|---|
| 企業ID | 営業リストの企業IDと紐付け | 主キー |
| 会社名 | 参照用 | 営業リストと重複保持（表示の簡便化のため） |
| 事業内容 | AI解析結果 | |
| 特徴 | AI解析結果 | |
| ターゲット層 | AI解析結果 | |
| 強み | AI解析結果 | |
| キーワード | AI解析結果（カンマ区切り） | |
| 解析元URL | 実際にAIへ渡した元データのURL | |
| 解析日時 | 実行日時 | |
| モデル名 | `gemini-2.0-flash` 固定（将来のモデル変更に備えて列として保持） | |
| 解析ステータス | 成功 / 失敗 | |
| 解析エラー内容 | 失敗時の理由 | |
| 下書き件名 | 直近生成分のみ（上書き） | |
| 下書き本文 | 直近生成分のみ（上書き） | |
| 下書きトーン | 生成時に選択したトーン | |
| 下書き生成日時 | | |
| 下書きステータス | 成功 / 失敗 | |
| 下書きエラー内容 | 失敗時の理由 | |

下書き用に別シートを設けず、企業分析シートに列を追加する形にすることで、④（履歴を持たずシンプルにする）と管理対象シート数の増加抑制を両立する。

### 営業リストシートへの追加列

既存19列（設計書④参照）に加え、20列目として以下を追加する。

| 列名 | 内容 |
|---|---|
| AI要約 | 企業分析結果の事業内容を1文程度に要約したもの（一覧表示用。詳細は企業分析シート・インラインパネル側を参照） |

`analyzeCompanyInfo()` 実行時に、企業分析シートへの保存とあわせてこの列も同時更新する。

## ③ Apps Script構成・API一覧

### 新規 Service: `GeminiService.js`

| 関数 | 概要 |
|---|---|
| `callGemini_(prompt, options)` | Gemini API (`generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent`) をUrlFetchAppで呼び出す共通関数。JSON構造化出力（`responseMimeType: "application/json"` + `responseSchema`）を利用し、パース済みオブジェクトを返す。APIキーはスクリプトプロパティ`GEMINI_API_KEY`から取得 |
| `buildAnalysisPrompt_(companyName, websiteText)` | 企業情報解析用のプロンプトを組み立てる |
| `buildDraftPrompt_(analysis, companyName, tone)` | 営業メールDM下書き生成用のプロンプトを組み立てる |

### 新規 Service: `AiAnalysisService.js`

| 関数 | 概要 |
|---|---|
| `getCompanyAnalysisRow_(companyId)` | 企業分析シートから該当企業の行を取得する |
| `upsertCompanyAnalysis_(companyId, companyName, analysisResult)` | 解析結果を企業分析シートへupsertする |
| `upsertDraft_(companyId, draftResult, tone)` | 下書き結果を（同じ行の）下書き列へ上書き保存する |
| `updateCompanySummary_(companyId, summaryText)` | 営業リストシートの「AI要約」列を更新する |
| `extractVisibleText_(html)` | HTMLからタグ・スクリプトを除去し、Geminiに渡すプレーンテキストを抽出する（簡易版） |

### Router層 追加関数

| 関数 | 概要 |
|---|---|
| `getCompanyAiPanel(companyId)` | インラインパネル表示用に、企業分析結果＋下書きをまとめて返す（未解析ならその旨のステータスを返す） |
| `analyzeCompanyInfo(companyId)` | Webサイトを解析し、Geminiで事業内容・特徴・ターゲット層・強み・キーワードを生成、企業分析シート＋営業リストのAI要約列を更新して結果を返す |
| `generateSalesDraft(companyId, tone)` | 保存済みの企業分析結果をもとに営業メールDM下書き（件名・本文）を生成し、上書き保存して返す |

## ④ 処理フロー

```
[営業リスト一覧]
    │ 「AI分析」トグルをクリック
    ▼
getCompanyAiPanel(companyId) 呼び出し
    │
    ├─ 未解析 → 「解析を実行する」ボタンのみ表示
    └─ 解析済み → 事業内容等＋（あれば）下書きを表示

[解析を実行する] クリック
    ▼
analyzeCompanyInfo(companyId)
    │ 1. 営業リストから対象企業のホームページURLを取得
    │ 2. UrlFetchAppでHTML取得（Sprint2のWebサイト解析と同様の取得処理を再利用）
    │ 3. extractVisibleText_() でタグ除去しプレーンテキスト化
    │ 4. buildAnalysisPrompt_() でプロンプト作成
    │ 5. callGemini_() でGemini APIへ送信、構造化データを受信
    │ 6. upsertCompanyAnalysis_() で企業分析シートへ保存
    │ 7. updateCompanySummary_() で営業リストのAI要約列を更新
    ▼
結果をインラインパネルに反映

[たたき台を生成する] クリック（トーン選択可）
    ▼
generateSalesDraft(companyId, tone)
    │ 1. 企業分析シートから解析結果を取得（未解析ならエラーを返す）
    │ 2. buildDraftPrompt_() でプロンプト作成
    │ 3. callGemini_() で件名・本文を生成
    │ 4. upsertDraft_() で下書き列へ上書き保存
    ▼
件名・本文をインラインパネルのテキストエリアに表示（コピー用ボタンあり）
```

## ⑤ 営業リストとの連携方法

- 主キーは既存の「企業ID」で統一し、企業分析シートも営業リストシートも同じ企業IDで紐付ける
- 一覧画面の「AI要約」列により、AI分析を開かなくても概要を把握できる
- 詳細（事業内容・特徴・ターゲット層・強み・キーワード・下書き）はインラインパネルでのみ表示し、シート構成をシンプルに保つ
- 将来的に企業詳細画面を実装する際は、このインラインパネルの内容をそのままタブとして移植できる構成にしておく

## ⑥ 実装ステップ（小さく分割・各Step末にデプロイ＆動作確認）

| Step | 内容 | 完了条件 |
|---|---|---|
| **Step1** | Gemini APIキー設定・疎通確認 | `GEMINI_API_KEY`スクリプトプロパティを設定し、`GeminiService.callGemini_()`で固定プロンプトを送信→レスポンスが正しく返ることを確認する（デバッグ用の最小実行のみ、UI組み込みなし） |
| **Step2** | 企業情報解析（事業内容・特徴・ターゲット） | 企業分析シートのスキーマ作成、`AiAnalysisService.js`のCRUD実装、`analyzeCompanyInfo()`実装。最小限の動作確認用UI（既存画面への簡易ボタン等）で1社解析できることを確認する |
| **Step3** | 営業メール・DM生成 | `generateSalesDraft()`実装、下書き列への上書き保存。Step2と同様、最小限のUIで1社分の下書き生成ができることを確認する |
| **Step4** | 営業リスト連携（要約列・インライン表示） | 営業リストシートへの「AI要約」列追加、`CompanyList.html`へのインラインアコーディオンUI実装、Step2・Step3の機能をこのUIに統合。一覧上でAI要約が見え、行を展開してStep2/3の操作が行えることを確認する |

ご相談いただいた内容のうち、元のご指示では「Step3」が2回登場していたため、上表のとおりStep1〜4として振り直しています。

## ⑦ 未確定・要確認事項

- `GEMINI_API_KEY`の取得（Google AI StudioでのAPIキー発行。管理者による作業が必要）
- Gemini APIの利用料金・レート制限（1社あたりの解析・生成コスト、無料枠の範囲）
- Webサイトのテキスト量が多い場合のプロンプト長・トークン数の上限対応（極端に大きいサイトは先頭N文字程度に切り詰める想定）
- 下書きのトーン選択肢の具体的なラインナップ（初回アプローチ／フォローアップ等、初期実装では最小限からスタートし後で拡張）
