# ADR-002: 企業ID の形式を C000001 形式に統一する

**ステータス**: Accepted  
**日付**: 2026-07-01（Issue #022）  
**決定者**: seki@innovation-zero.co.jp

---

## 背景

企業の識別子として以下の変遷があった：

1. **初期**: Google Place ID（`ChIJ...`）をそのまま ID として利用
2. **Issue #021**: `place:ChIJ...` 形式 → 人が読める連番（`000001`）へ移行
3. **Issue #022**: `000001` 形式 → `C000001` 形式へ移行（本 ADR の対象）

Issue #022 での変更理由は、将来的に「案件ID」「担当者ID」などの異なるエンティティを
同一シートまたは関連シートで管理する際に、ID の種別を目視で判断できる必要が生じたため。

---

## 判断

**企業ID は `C000001` 形式（大文字アルファベット 1 文字 + 6 桁ゼロ埋め数字）に統一する。**

将来の拡張を見据えて `ID_PREFIXES` 定数と `formatId(prefix, n)` 共通関数を導入し、
新しいエンティティ種別は 1 行追加するだけで対応できる設計とする。

---

## 理由

- プレフィックスにより、シートを目視したときに `C000001` が企業、`D000001` が案件だと即座に分かる
- 純数字（`000001`）は将来の案件ID との衝突・混同リスクがある
- 6 桁ゼロ埋めによりソート時の並び順が番号順になる（アルファベット順と数値順が一致する）

---

## 却下した選択肢

**UUID / nanoid を使う案**

グローバルユニーク性は高いが、シートを手動で見たときに無意味な文字列になる。
営業担当者がメモに「C000012 の件」と書けることが重要なため却下。

**プレフィックスなし連番（000001）を継続する案**

シンプルだが、将来のエンティティ拡張時に種別の識別手段がなくなる。

---

## 影響

**マイグレーション**

- `000001` 形式（Issue #021 以前）は `C000001` へ自動変換（番号を保持）
- `place:ChIJ...` 形式（Issue #021 以前）は新規連番を割り当て、Place ID は専用フィールドへ保存
- マイグレーションスクリプト: `scripts/migrate-company-ids.mjs`

**不変性の保証**

- 採番後の `companyId` は `PROTECTED_FIELDS` に含まれ、ツールによる自動上書きを禁止
- `updateStatus` の `expectedCompanyId` オプションによりソート後の行番号ずれを自動補正

**拡張予定のプレフィックス**

```js
export const ID_PREFIXES = Object.freeze({
  company: 'C',
  // deal:   'D',   // 案件ID（将来予定）
  // person: 'P',   // 担当者ID（将来予定）
});
```

**関連実装**

- `src/sheets/mapper.js` — `ID_PREFIXES`、`formatId`、`formatCompanyId`、`parseCompanyId`
- `src/sheets/service.js` — `_resolveCompanyIdConflicts`（並列書き込み時の ID 衝突解消）
