## 次期開発計画（v0.9.x）— 2026-07-02 レビューによる再設計

**再設計の背景**: #027 完了時点でのレビューにより、「送信するまで」は CLI だけで完結する一方、
「送信した後」（返信 → 商談 → 成約/失注）は一切 CLI 化されておらず、シートを開いた手入力に
依存していることが判明した。ツールの核心（シートを開かなくていい）が営業として最も重要な
局面で崩れているため、**まず今の業務が CLI だけで回り切るようにする**ことを優先する。
ただし複数テンプレート管理は営業現場での利用頻度が高いため、レビュー時点の初期案から
前倒しし、leadScore 活用・AI パーソナライゼーションより先に着手する。

### Issue #028: ステータス更新 CLI（`update`）+ 失注ステータス正式導入

**目的**: 返信後のライフサイクル（商談日設定・成約・失注・メモ）をシートを開かずに CLI だけで完結させる。#027 で「将来ステータス追加予定」として 0 件固定にした失注をここで実装する。

**実装内容**

- 新規コマンド `sales-pilot update <companyId>`
- オプション: `--meeting <YYYY-MM-DD>`（商談日設定） / `--won`（成約） / `--lost [<理由>]`（失注、理由はメモへ追記） / `--memo <text>`（メモ追記）
- `closed`（成約）フィールドの値域を `''` / `'成約'` / `'失注'` の enum に拡張し、失注を正式な状態として持てるようにする（既存の成約データは後方互換のマイグレーションで吸収）
- `SheetsService.updateCompanyByCompanyId()` を再利用（新規 API コール追加なし）
- `SheetsService._ensureColumnsExist()`（新規）: 更新対象フィールドの列がシートに存在しない場合、企業ID / Place ID 列と同じパターンでヘッダー行の末尾に自動追加する。「商談日」「成約」列を持たない既存シートでも `update` がそのまま使える
- `SheetsService.getStats()` の `closedCount` / `lostCount` を実データから算出するよう更新

**変更ファイル**

- `src/cli/commands/update.js`（新規）
- `src/cli/commands/update.test.js`（新規）
- `src/sheets/service.js`（`_ensureColumnsExist()` 追加・`getStats()` の集計ロジック更新）
- `src/sheets/service.test.js`（新規）
- `src/sheets/mapper.js`（`closed` フィールドの値域）
- `src/cli/index.js`

**受入基準**: `sales-pilot update C000001 --lost "予算合わず"` でシートを開かずに更新でき、`sales-pilot report` の失注件数に反映される

### Issue #029: 今日のアクションリスト（`follow-up`）

**目的**: 「今日誰に何をすべきか」を一目で分かるようにする。#026 の元設計にあった「フォロー候補（送信から 7 日以上・返信なし）」を正式に実装する。

**実装内容**

- 新規コマンド `sales-pilot follow-up`
- 送信から N 日以上経過・未返信の企業を抽出（`sentDate` 基準、閾値は `--days <n>` で変更可、デフォルト 7）
- 本日が `meetingDate` の企業を抽出
- 送信承認待ち（`送信可否` 未設定）の新規リード件数を表示
- `getAllCompanies()` のみから算出し、新規 API コールは追加しない

**変更ファイル**

- `src/cli/commands/follow-up.js`（新規）
- `src/cli/commands/follow-up.test.js`（新規）
- `src/sheets/service.js`（`getFollowUpCandidates()` 追加）
- `src/cli/index.js`

**受入基準**: `sales-pilot follow-up` で「要フォロー」「本日商談」「承認待ち」の 3 リストが表示される

### Issue #030: `find --review` 対話的承認モード

**目的**: 検索結果をその場で確認・承認し、シートを開かずに送信可否まで確定させる（find → シート編集 → send の往復をなくす）

**実装内容**

- `--review` / `-r` フラグ
- 企業ごとに「会社名 / URL / 電話 / スコア」を表示
- `s`=保存 / `x`=スキップ / `o`=保存+送信承認 / `q`=終了
- 選択結果をまとめて一括 append

**変更ファイル**

- `src/cli/commands/find.js`
- `src/crawler/find.js`

**受入基準**: `find --review` で 1 件ずつ確認しながら `s/x/o/q` を選択でき、選択結果が一括保存される

### Issue #032: leadScore 活用 + 送信後ダッシュボード自動更新

**目的**: Provider が算出するだけで未使用だった `leadScore` を実際の優先順位付けに使い、`send` 実行後もダッシュボードが最新状態を保つようにする（`check-replies` は既に更新している）

**実装内容**

- `find` の結果を `leadScore` 降順でソートして保存・表示
- `find --min-score <n>`: リードスコア閾値フィルタ
- `send` を `leadScore` 降順で送信（優先度の高い企業から連絡）
- `send` 完了後に `DashboardService.createOrUpdateDashboard()` を呼ぶ

**変更ファイル**

- `src/crawler/find.js`
- `src/cli/commands/find.js`
- `src/cli/commands/send.js`

**受入基準**: スコアの高い企業ほど `find` の表示・`send` の送信順で先に来る。`send` 実行後にダッシュボードが更新される
