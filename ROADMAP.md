## 次期開発計画（v0.9.x）— 2026-07-02 レビューによる再設計

**再設計の背景**: #027 完了時点でのレビューにより、「送信するまで」は CLI だけで完結する一方、
「送信した後」（返信 → 商談 → 成約/失注）は一切 CLI 化されておらず、シートを開いた手入力に

### Issue #029: 今日のアクションリスト（`follow-up`）

**目的**: 「今日誰に何をすべきか」を一目で分かるようにする。#026 の元設計にあった「フォロー候補（送信から 7 日以上・返信なし）」を、3日/7日の2段階・商談日は今日/明日を分離する形で正式に実装する。承認待ち新規リード件数は `report` 側の役割とし、ここでは扱わない。

**実装内容**

- 新規コマンド `sales-pilot follow-up`
- カテゴリ分類（優先順位順）:
  1. `meetingToday` — 商談日が今日
  2. `waitingUrgent` — 送信済・未返信・商談日未設定、経過 7 日以上
  3. `meetingTomorrow` — 商談日が明日
  4. `waitingWarning` — 送信済・未返信・商談日未設定、経過 3〜6 日
- 除外ルール: 成約/失注確定済み（`closed` が空でない）、配信停止の企業は対象外
- 商談日が設定済みの企業は「返信待ち」カテゴリから除外する（二重表示防止）
- 分類ロジックは `src/sheets/follow-up.js` の純粋関数（`buildFollowUpList(companies, referenceDate)`）に分離し、I/O を持たない。CLI・将来の Web管理画面・通知機能（Slack/LINE）から共通利用できる Service 層として設計する
- `SheetsService.getFollowUpList()`: `getAllCompanies()` の結果を上記純粋関数に委譲するだけの薄いラッパー（新規 API コール追加なし）
- 各項目に `actionType`（`MEETING` / `FOLLOW_UP` / `WAIT_REPLY`）と `priority` を付与し、Web 側でのアイコン・色分けに利用できるようにする
- `getAllCompanies()` のみから算出し、新規 API コールは追加しない

**変更ファイル**

- `src/sheets/follow-up.js`（新規、純粋関数）
- `src/sheets/follow-up.test.js`（新規）
- `src/sheets/service.js`（`getFollowUpList()` 追加）
- `src/sheets/service.test.js`（委譲テスト追加）
- `src/sheets/index.js`（`buildFollowUpList` 等のエクスポート追加）
- `src/cli/commands/follow-up.js`（新規）
- `src/cli/commands/follow-up.test.js`（新規）
- `src/cli/index.js`

**受入基準**: `sales-pilot follow-up` で「本日の商談予定」「要フォロー（7日以上）」「明日の商談予定」「経過観察（3日以上）」の4カテゴリが優先順位順に表示される

### Issue #030: `find --review` 対話的承認モード

**目的**: 検索結果をその場で確認・承認し、シートを開かずに送信可否まで確定させる（find → シート編集 → send の往復をなくす）

**実装内容**

- `--review` / `-r` フラグ
- 企業ごとに「会社名 / URL / 電話 / スコア」を表示
- `s`=保存 / `x`=スキップ / `o`=保存+送信承認 / `q`=終了
- 選択結果をまとめて一括 append

**変更ファイル**

- `src/cli/commands/find.js`
- `src/crawler/find.js`

**受入基準**: `find --review` で 1 件ずつ確認しながら `s/x/o/q` を選択でき、選択結果が一括保存される

### Issue #032: leadScore 活用 + 送信後ダッシュボード自動更新

**目的**: Provider が算出するだけで未使用だった `leadScore` を実際の優先順位付けに使い、`send` 実行後もダッシュボードが最新状態を保つようにする（`check-replies` は既に更新している）

**実装内容**

- `find` の結果を `leadScore` 降順でソートして保存・表示
- `find --min-score <n>`: リードスコア閾値フィルタ
- `send` を `leadScore` 降順で送信（優先度の高い企業から連絡）
- `send` 完了後に `DashboardService.createOrUpdateDashboard()` を呼ぶ

**変更ファイル**

- `src/crawler/find.js`
- `src/cli/commands/find.js`
- `src/cli/commands/send.js`

**受入基準**: スコアの高い企業ほど `find` の表示・`send` の送信順で先に来る。`send` 実行後にダッシュボードが更新される
